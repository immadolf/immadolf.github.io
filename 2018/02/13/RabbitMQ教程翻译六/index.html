    Remote procedure call (RPC)

    RPC（远程过程调用）

    在第二章中我们学习了如何使用工作队列在多个工作线程中分发耗时的任务。

    但是如果我们需要调用远程计算机上的某个函数并等待它的结果呢？这种形式被称为远程方法调用。

    在这一节中，我们将使用RabbitMQ构建一个RPC系统：一个客户端和一个可扩展的RPC服务端。我们将通过一个远程的求斐波那契数列的服务来完成示例。

    客户端接口

    我们会创建一个简单的客户端类来阐释RPC服务是如何被调用的。这个类有一个名为call的方法用来发送一个RPC请求并在结果返回之前保持阻塞。

    虽然RPC在现代计算中是非常常见的模式，但它仍然饱受批评。问题出现在程序员往往不清楚一个函数调用是本地调用还是远程方法调用。这样会导致系统的结果变得不可预知，而且也增加了调试的复杂度。滥用RPC不仅不能简化代码还会使得代码变得难以维护。

    考虑到这一点，请参考以下建议：

    • 清楚的知道哪些方法是本地调用哪些方法是远程调用
    • 为你的系统添加文档，使每个组件之间的依赖关系清晰可见。
    • 处理各种异常。如果RPC服务器长时间宕机，客户端应该如何处理。

    如果存在疑问，请避免使用RPC。如果可以则应使用异步的管道而不是阻塞的RPC。

    回调队列

    总的来说在RabbitMQ上进行RPC调用是比较容易的。客户端发送了一个请求消息，服务端返回一个响应消息。为了接收响应信息，我们需要在请求信息中加入回调队列的信息。

    Message Properties

    AMQP协议预定义了14个属性，大多数的属性平时很少用到，下面是比较常见的几个属性：

    • deliverMode：当值为2时，标记消息为持久化，而其它值时，消息为非持久化。
    • contentType：描述编码的mime-type，对于常见的json编码来说设置这个属性为application/json是个好的实践。
    • replyTo：通常用来指定回调队列的名字。
    • correlationId：有助于将RPC响应与请求关联起来

    关联Id(Correlation Id)

    在上面介绍的方法中，我们为每个RPC请求创建了一个回调队列。这是非常低效的，但是有一个更好的方法就是为每个客户端创建一个回调队列。

    这就有了一个新的问题，监听回调队列中的响应消息的方法不知道哪个响应是属于它的。这个时候就需要使用correlationId了。我们为每个请求设置一个独一无二的correlationId，这样当收到回调队列中的消息时只需要查看correlationId是否与自己匹配即可，如果不匹配，则不处理。

    总结

    我们的RPC是这样工作的：

    1. 当客户端启动的时候创建一个匿名独占的回调队列
    2. 对于一个RPC请求，客户端会设置两个值：replyTo指定回调队列，correlationId指定一个独一无二的值。
    3. 请求将被发往rpc_queue队列
    4. 服务端等待队列中的请求。当有请求到达时，服务端处理请求然后将结果发往请求中标记的回调队列。
    5. 客户端等待回调队列的消息。当有消息到达时，检查其中的correlationId是否匹配，如果匹配则返回结果给应用。
