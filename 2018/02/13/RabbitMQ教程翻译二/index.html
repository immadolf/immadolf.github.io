    Work queues

    工作队列

    在上一节中，我们编写了一个往特定队列发送消息和一个从特定队列接收消息的程序。在这一节中，我们会创建一个在多个消费者之间分发耗时任务的工作队列。

    工作队列（或者说任务队列）背后的主要思想是避免立即执行资源密集型计算以至于不得不等待它完成。与通过调度任务让它稍后执行不同，我们把一个任务封装成一条消息然后把它放进一个队列中。在后台运行的工作线程将会从队列中取出这个任务且最终将执行这个任务。当后台有很多的工作线程时，它们之间将共享任务。

    准备

    在前一节的教程中，我们发送了一条包含了"Hello World"的消息。现在我们将发送一条代表复杂任务的字符串。在现实应用中，复杂任务可以是对一个图片做resize，或者是渲染pdf文件或内容提取。但是作为Demo，我们通过Thread.sleep()来模拟复杂任务。我们用字符串中.的数量来表示任务的复杂度，多一个.就表示任务的耗时多一秒。例如，Hello...将会耗时三秒。

    我们稍微修改一下前一节中的Send.java以使它支持发送命令行中输入的任意字符串。这个程序将会调度任务到工作队列中，因此我们给它一个新的名字NewTask.java。

    原先的Recv.java程序也需要做些修改：对每个消息体中的.，它都需要假装有一秒的工作要处理。因为它会处理收到的消息并执行任务，因此我们称它为Worker.java。

    循环分发

    使用工作队列（任务队列）的一个优点是可以轻松的并行工作。如果积压了大量的工作，我们只需要增加更多的工作线程，这样就可以轻松地扩大规模。

    消息确认

    执行一个任务要花费一定的时间。如果一个消费者在这段时间内异常退出了将会如何？在我们现有的代码中，RabbitMQ在将消息分发到消费者时就会将消息标记为删除。在这种情况下，我们就丢失了这一条消息。同时那些被分到这一个消费者但是还没有来得及处理的消息也会被丢失。

    但是我们肯定不希望丢失任何任务。如果一个消费者异常退出了，我们希望这个任务会被分发到其他存活着的消费者。

    为了确保消息不会被丢失，RabbitMQ提供了消息确认机制。消费者会发送一个ack(nowledgement)给RabbitMQ通知它这个消费者已经接收到并处理完分发给它的某个消息，然后RabbitMQ才可以删除这条消息。

    如果一个消费者异常退出时还没有来得及发送ack，RabbitMQ会理解成这个消息没有被处理完，然后将这个消息重新放进消息队列中。如果有存活着的消费者，这条消息会被快速分发到其它存活着的消费者。这样就可以确保即使是消费者异常退出时也不会丢失消息。

    因为这里没有设置超时时间，因此RabbitMQ只有当某个消费者异常退出时才会重分发消息。这适用于处理非常耗时的任务。

    忘记ack是一个很常见的错误，但是后果却非常严重。当消费者退出时，消息会被重新分发。然后RabbitMQ会因为不能删除那些未被确认的消息而占用越来越多的内存。可以通过rabbitmqctl命令来调试这个问题，这个命令可以打印出未被确认的消息：

    sudo rabbitmqctl list_queue name messages_ready messages_unacknowledged

    公平分发

    你可能已经注意到了分发机制并不像我们想象的那种工作。例如在两个消费者的情况下，当所有的奇数消息都非常耗时而偶数消息都很轻量时就会出现一个消费者工作很重以至于没有时间休息而另一个消费者却无所事事的现象。

    发生这个情况的原因是RabbitMQ仅仅分发队列中的消息，而不看每个消费者还有多少未ack的消息。也就是RabbitMQ的默认机制是将第n个消息分发给第n个消费者。

    我们可以将channel的basicQos设为1以应对上述情况。这个参数的意思是告诉RabbitMQ在一个消费者没有返回ack之前不要给它分发新的消息。

    当所有的消费者都很繁忙时，消息队列就有可能会被填满，因此需要留心这种情况，或者可以添加更多的消费者，或者可以使用其他的策略。
